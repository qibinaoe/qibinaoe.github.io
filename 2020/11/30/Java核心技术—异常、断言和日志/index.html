Java核心技术——异常、断言和日志

发表于 2020-11-30 更新于 2022-06-08 分类于 Java 阅读次数： 本文字数： 5.5k 阅读时长 ≈ 9 分钟

Java核心技术 第七章 异常、断言和日志

处理错误

Java使用异常处理(exception handing)的错误捕获机制

异常处理的任务是将控制权从产生错误的地方转移到能够处理这种情况的错误处理器

错误的问题

  • 用户输入错误
  • 设备错误
  • 物理限制
  • 代码错误

如果方法不能够以正确的途径完成任务，可以抛出(throw)一个封装了错误信息的对象。这个方法会立刻退出，并不返回正常值(或任何值)。同时异常处理机制开始搜索能够处理这种异常状况的异常处理器(exception handler)

异常对象

所有异常都是派生于Throwable类的一个类实例

  • Throwable
    • Error
    • Exception
      • 其他异常(例如IOException)
      • RuntimeException

Throwable分成Error和Exception。

Error类层描述Java运行时，系统的内部错误和资源耗尽错误。我们自身的程序不应该抛出这种类型对象。出现这种内部错误应该通知用户和妥善终止程序。

Exception层分成两个分支：一个派生于RuntimeException，另一个分支包含其他异常。

由编译错误导致的异常属于RuntimeException

  • 错误的强制类型转换
  • 数组访问越界
  • 访问null指针

如果出现RuntimeException异常，那么就一定是你的问题

程序本身没有问题，由于像I/O错误这类问题导致的异常属于其他异常

不是派生于RuntimeException类的异常

  • 试图超越文件我末尾继续读取数据
  • 试图打开一个不存在的文件
  • 试图根据给定的字符串查找Class对象，而这个字符串表示的类不存在

非检查型(unchecked)异常

  • 派生于Error类
  • 派生与RuntimeException类

检查型(checked)异常

  • 所有其他的异常

编译器将检查是否为所有的检查型异常提供了异常处理器

方法不仅需要告诉编译器将要返回什么值，还要告诉编译器有可能发生什么错误

要在方法的首部指出这个方法可能抛出一个异常，以反映这个方法可能抛出的检查型异常 public FileInputStream(String name) throws FileNotFoundException

如果真的抛出了一个异常对象，运行时，系统就会开始搜索知道如何处理FileNotFoundException对象的异常处理器

检查型异常

会抛出异常的情况:

  • 调用一个抛出检查型异常的方法，如FileInputStream构造器
  • 检测到一个错误，并且利用throw语句抛出一个检查型异常
  • 程序出现错误，例如a[-1] = 0;会抛出一个非检查型异常(ArrayIndexOutOfBoundsException)
  • Java虚拟机或运行时库出现内部错误

异常规范(exception specification)声明

一个方法可能抛出多个检查型异常类型，在方法首部列出所有的异常类，每个异常类之间用逗号隔开

  • 一个方法必须声明所有可能抛出的检查型异常
  • 非检查型异常
    • 不需要声明Java的内部错误，即从Error继承的异常
      • 在我们控制之外
    • 不应该声明从RuntimeException继承的非检查型异常
      • 这种错误在我们控制中，应该从一开始去避免。

如果子类覆盖超类的第一个方法

  • 子类方法中声明的检查型异常不能比超类方法中声明的异常更通用
    • 子类可以抛出更特定的异常或者不抛出异常
  • 超类没有抛出任何检查型异常，子类也不能抛出任何检查型异常

抛出的异常实例可能属于声明的异常的类，也可能属于这个类的任意一个子类

抛出异常的方式

  • 找到一个合适的异常类
  • 创建这个类的一个对象
  • 将对象抛出

创建异常类

情况：遇到任何标准异常类都无法描述清楚的问题

派生于Exception的类或派生于Exception的某个子类

写两个构造器：一个是默认的构造器，一个是包含详细描述信息的构造器

捕获异常

捕获异常

如果发生异常，但没有在任何地方捕获这个异常，储层徐就会终止，并在控制台上打印这个异常的类型和一个堆栈轨迹

捕获异常需要设置try/catch语句块

  • 如果try语句块中任何代码抛出了catch子句指定的一个异常类
    • 程序将跳过try语句块的其余代码
    • 层序执行catch子句中处理器代码
  • 如果try语句块中的代码没有抛出任何异常，那么程序跳过catch子句
  • 如果方法中任何代码抛出了catch子句中没有声明的一个异常类型
    • 方法立即退出

如果编写一个方法覆盖超类的方法，而超类方法中没有抛出异常，则必须捕获方法代码中出现的每一个检查型异常。即不允许子类的throws说明符中出现超类方法未列出的异常类

捕获多个异常

对不同类型的异常做出不同的处理为每个异常类型使用一个单独的catch子句，在Java 7中，同一个catch子句中可以捕获多个异常

  • e.getMessage 获得更多信息

  • e.getClass().getName() 异常对象的实际类型

捕获多个异常时，异常变量隐含为final变量

捕获多个异常会让代码更简单和高效

再次抛出异常与异常链

finally 子句

不管是否有异常被捕获，finally子句的代码都会执行

finally子句用于处理资源分配后的清理

try语句可以只有finally子句而没有catch子句

情况分析

  • 代码没有抛出异常，程序首先执行try语句块中全部代码，然后执行finally子句中的代码，再执行finally子句之后的第一条语句
    • 即 1、2、5、6
  • 代码抛出异常并在catch中捕获
    • try中执行到语句发生异常的位置，跳过try中的剩余部分代码，在一个catch子句中捕获。执行catch子句中的代码，最后执行finally子句中的代码
      • 如果catch子句没有抛出异常，则将执行finally子句后的第一条语句
        • 执行顺序为1、3、4、5、6
      • 如果catch子句抛出异常，则异常将抛回这个方法的调用者
        • 执行顺序为1、3、5
  • 代码抛出了一个异常，但没有任何catch子句捕获这个异常。程序执行try语句块的语句直到抛出异常位置，然后跳过try语句块中剩余代码，然后执行finally子句的语句，并将异常返回给这个方法的调用者
    • 执行顺序1、5
  • 内层try语句块时确保关闭输入流
  • 外层try语句块时确保报告出现的错误

finally子的体，用于清理资源。不要把改变控制流的语句(return, throw, break, continue)放在finally子句中

try-with-resources (Java 7新特性)

try-with-resources语句(带资源的try语句)的最简形式

try块退出时，会自动调用res.close();

try()中如果只有一个资源，不用分号。如果有多个资源，最后一个资源后面不用分号。

try-with-resources能够自动清理资源，但是还是要写throws或者catch语句

使用异常的技巧

  • 异常处理不能代替简单的判断
    • 基本规则：只在异常情况下使用异常
  • 不要过分地细化异常
    • 极端例子：每个句子都写一个try
    • 承诺：将正常处理与错误处理分开
  • 充分利用异常的层次结构
    • 应该寻找一个适合的子类或创建自己的异常类
    • 不要为逻辑错误抛出异常
    • 能够将一种异常转换成另一种更加合适的异常，那么不要犹豫。
  • 不要压制异常
  • 在检测错误时，“苛刻”比放任更好
  • 不要羞于传递异常

断言

关键字assert

形式

  • assert conditon

  • assert condition : expression

Java中3种处理系统错误的机制

  • 抛出一个异常
  • 日志
  • 使用断言

使用断言的场景

  • 断言失败是致命的、不可恢复的错误
  • 断言检查只是在开发和测试阶段打开
