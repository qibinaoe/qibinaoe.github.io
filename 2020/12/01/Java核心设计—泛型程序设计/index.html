Java核心设计 第八章 泛型程序设计

泛型程序设计

泛型设计(generic programming) : 编写的代码可以对多种不同类型的对象重用

类型参数(type parameter) ArrayList<String> files = new ArrayList<>();

在Java增加泛型类之前，泛型程序设计是用继承实现的，存在两个问题

  • 当获取一个值时，必须进行强制类型转换
  • 没有错误检查，可以向数组列表中添加任何类的值

在使用泛型类和类型参数后

  • 编译器可以通过类型参数来知道返回值的类型，因此不需要进行强制类型转换
  • 在添加元素（ArrayList中的add()）时，编译器知道插入元素的类型，比Object类型安全。编译器可以检查，防止插入错误类型的对象

通配符类型(wildcard type)

泛型类

泛型类(generic class) :有一个或多个类型变量的类

  • 类型变量 T ，用尖括号(<>)括起来，放在类名后面

  • 泛型类可以有多个类型变量 public class Pair<T,U> {...}

  • 类型变量在整个类定义中，用于

    • 指定方法的返回类型
    • 字段的类型
    • 局部变量的类型

类型变量使用大写字母

  • 变量E表示集合的元素类型
  • K和V分别表示表的键和值的类型
  • T（和相邻的字母U和S）表示”任意类型“

泛型类相当于普通类的工厂

  • 用具体的类型替换类型变量来**实例化(instantiate)**泛型类型

泛型方法

泛型方法可以在普通类中定义也可以在泛型类中定义

**类型变量**的位置

  • 放在修饰符（这里指public static）的后面
  • 放在返回类型的前面（这里的返回类型是T）

调用一个泛型方法

  • 可以把具体类型包为在尖括号中，放在方法名前面
    • String middle = ArrayAlg.<String>getMiddle("Jogn","CAt","Dog");
  • 大多数情况下，编译器有足够信息推断出想法，将参数的类型和泛型类型T进行匹配进行推断，可以省略<String>类型参数
    • String middle = ArrayAlg.getMiddle("Jogn","CAt","Dog");

类型变量的限定

有时，类或方法需要对类型变量加以约束(例如保证方法中的某些方法能够正常调用)，可以通过对类型变量T设置一个**限定(bound)**来实现

使用关键字extends

记法 <T extends BoundingType>

  • T表示限定类型（BoundingType）的子类型（subtype）
  • T和限定类型可以是类也可以是接口
  • 一个类型变量(T)或通配符可以有多个限定(Bound)
    • T extends Comparable & Serializable
    • 限定类型（BoundingType）用”&”分隔
    • 类型变量（T)用逗号分隔
  • 在限定类型中可以拥有多个接口作为超类型，但最多有一个限定可以是类。
    • 如果一个类作为限定，它必须是限定列表中的第一个限定

泛型代码与虚拟机

  • 虚拟机没有泛型类型对象，所有对象都属于普通类

类型擦除

  • 无论何时定义一个泛型类型，都会自动提供一个相应的原始类型（raw type）。
    • 原始类型的名字就是去掉类型参数后泛型类型名
  • 类型变量会被擦除（erased），并替换为其限定类型（对于无限定的变量替换成Object）

Pair<T>的原始类型（raw type）

这里，T是无限定的变量，所以直接用Object替换

C++会为每个模板的实例化产生不同的类型，这一现象叫做“模板代码膨胀”。Java不存在这个问题

原始类型用第一个限定来替换类型变量。如果没有给定限定，则替换为Object

为提高效率，应该将标签（tagging）接口（即没有方法的接口）放在限定列表的末尾。

方法擦除的两个问题：

  • 类型擦除与多态发生冲突

例如继承泛型类的子类中，重写父类的方法。再类型擦除后，会有一个从父类继承来的方法，一个自己重写的方法

在类型擦除后，有两个同名方法

第二个是从父类继承来的。

解决办法是在子类中（这里指DateInterval类）生成桥方法（bridge method）

  • 类型擦除后，子类会有两个方法名和参数类型相同，但是返回类型不相同的方法

在虚拟机中，会有参数类型和返回类型共同制定一个方法，因此编译器可以为两个仅返回类型不同的方法生成字节码，虚拟机能够正确处理这种情况

桥方法也用在有协变的返回类型（convariant return type）：一个方法覆盖另一个方法时，可以指定一个更严格的返回类型。

  • 虚拟机没有泛型，只有普通的类和方法
  • 所有的类型参数都会替换为它们的限定类型
  • 会合成桥方法来保持多态
  • 为保持类型安全性，必要时会插入强制类型转换

使用注解（annotation）来消除警告

  • 给局部变量加注解
  • 给整个方法加注解

泛型的限制与局限性

  • 不能用基本类型实例化类型参数

    • 没有Pair<double>，只有Pair<Double>
    • 类型擦除后，Object类型的字段不能储存double值

  • 运行时类型查询只适用于原始类型

    • 虚拟机对象总有一个特定的非泛型类型，所有的类型查询只产生原始类型

    • 试图查询一个对象是否属于某个泛型类型(instanceof)，会得到编译器错误

    • 试图强制类型转换成泛型类型时，会得到的警告

    • if(a instanceof Pair<String>) // ERROR

if(a instanceof Pair<T>) //ERROR

Pair<String> p = (Pair<String>) a; // warning

Pair<String> stringPair = ...;
Pair<Employee> employeePair = ...;
if(stringPair.getClass() == employeePair.getClasss()) // equal 因为返回的都是原始类型




- 在向参数个数可变的方法传递一个泛型类型的实例时，Java虚拟机必须建立一个泛型数组。相对于上一条"不能创建参数化类型的数组"而言，这种情况会有所放松，会得到一个警告而不是错误
- 抑制警告的方法有
  - 方法增加注解`SuppressWarnings("unchecked")`
  - 或方法增加注解`SafeVaragrs`
    - `SafeVarargs`注解只能用于声明为`static`、`final`的方法（其他方法可能被覆盖，使得注解没有意义）

  • 不能实例化类型变量

    • 不能在类似 new T()的表达式中使用类型变量

      • public Pair() {first = new T(); second = new T();} // ERROR

    • 解决办法1：提供一个构造器表达式

      • Pair<String> p = Pair.makePair(String::new);

    • 解决办法2：反射

      • public static <T> Pair<T> makePair(Class<T> cl)
{
    try{
        return new Pair<>(cl.getConstructor().newInstance(),cl.getConstructor().newInstance());
    }
    catch(Exception e) {return null;}
}

  • 不能构造泛型数组

    • String[] names = ArrayAlg.minmax(String[]:: new,"Tom","Dick","Harry");

    • public static <T extends Comparable> T[] minmax(T... a)
{
    T[] result = (T[]) Array.newInstance(a.getClass().getComponentType(),2);
    ...
}

  • 不能抛出或捕获泛型类的实例

    • 泛型类扩展Throwable是不合法的

      • 不能定义一个泛型类，该类继承Throwable‘

      • public class Problem<T> extends Exception{...} //ERROR

    • 在异常规范中使用类型变量是允许的

      • 我理解是在 try-catch语句中，可以出现类型变量。但是不能在catch后面的小括号声明异常类型是T(类型变量)，也不能throw出一个是类型变量的异常的实例

      • 合法示例

    • 可以取消对检查型异常的检查

      • Java异常处理的基本原则：必须为所有检查型异常提供一个处理器（try-catch）

      • 通过泛型可以取消这种机制

      • 实现一个Task到Runnable的适配器，不过Runnable接口的类的run方法不允许抛出检查型异常，而我们要实现Task中的run可以抛出任何异常

  • 注意擦除后的冲突

    • 擦除后，方法的参数类型变成Object，可能会和产生相同签名的方法

      • 解决办法是修改方法名

    • 泛型规范说明原则之一

      • 为了支持擦除转换，施加一个限制：倘若两个接口类型是同一接口的不同参数化，一个类或类型变量就不能同时作为这两个接口类型的子类

      • 非法示例

        • class Employee implements Comparable<Employee> {...}
        • class Manager extends Employee implements Comparable<Manager>{...} // ERROR
          • 因为Comparable<Employee>和Comparable<Manager>是同一接口的不同参数化

      • 非泛型版本的合法示例

        • class Employee implements Comparable {...}
        • class Manager extends Employee implements Comparable{...}

泛型类型的继承规则

Employee和Manager的关系是父类和子类，但是Pair<Manager>不是Pair<Employee>的子类。

通常Pair<Employee>和Pair<Manager>没有任何关系

泛型类型不能够转换但是Java数组可以转换

泛型类可以扩展或实现其他的泛型类

  • ArrayList<T>类实现了List<T>接口
  • ArrayList<Manager>可以转换成一个List<Manager>

对于上面这两句话的理解可以看下面的代码。

超类型限定允许写入的意思是，由于超类变量可以引用其子类对象。因此我们可以声明我们要储存的位置为我们要真实要存放元素的超类。即generateCouple方法中体现的。想想如果我们改成Couple<? extends Human> c会发生什么情况。我们不能保证我们存进去c的元素就是Human类型，如果是Human的子类就会出错。

Human子类会出错的原因在于，我们不能保证是存进去的Human子类就是我们想要的类型。即Human可能有多个子类，例如Man和Woman，假如实际上我们声明的Couple中T的为Man，但是运行时候被传入一个Woman，就会出错。

同样的，对于子类型限定允许读取的意思是，在printLove方法中，我们能够保证我们要读取的元素c肯定有c.getHusband().getName()和c.getWife().getName()两种方法。因为子类会继承父类的所有可以继承的方法和实例。而如果改成超类型限定就不能够保证。

子类型限定另一个常见用法是作为一个函数式接口的参数类型
